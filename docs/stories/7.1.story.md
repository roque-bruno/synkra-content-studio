# Story 7.1 — Arquitetura de Armazenamento Seguro de API Keys

**Epico:** 7 — Configuracao Self-Service
**Status:** Done
**Executor:** @architect (Aria)
**Modo:** YOLO

---

## Descricao

Definir como API keys serao armazenadas de forma segura quando configuradas pela UI. Atualmente as credenciais ficam apenas no `.env` do servidor. Com a configuracao via UI, precisamos de um modelo que:
- Persista as keys de forma segura
- Nunca exponha keys completas ao frontend
- Sobreviva a restarts do container
- Integre com o sistema existente de env vars

## Acceptance Criteria

- [ ] ADR (Architecture Decision Record) documentando a abordagem escolhida
- [ ] Modelo de armazenamento definido (arquivo criptografado, SQLite, vault, etc.)
- [ ] Estrategia de migracao: keys existentes no .env continuam funcionando
- [ ] Hierarquia de prioridade: UI settings > .env (ou vice-versa, justificado)
- [ ] API keys nunca retornadas inteiras ao frontend (mascaramento obrigatorio)
- [ ] Modelo compativel com Docker volumes (persistencia)

## Scope

**IN:**
- Decisao arquitetural de storage
- Definicao de API de leitura/escrita
- Modelo de mascaramento para frontend
- Estrategia de migracao .env → UI

**OUT:**
- Implementacao (Stories 7.2-7.5)
- Criptografia at-rest com KMS externo (overengineering para MVP)

## Opcoes a Avaliar

| Opcao | Pros | Contras |
|-------|------|---------|
| JSON criptografado em volume | Simples, portavel | Chave de criptografia precisa de lugar seguro |
| SQLite com coluna encrypted | Queries faceis, backup simples | Overhead de DB para poucas keys |
| Arquivo .env.local gerado pela UI | Compativel com Docker | Menos seguro, plaintext |
| Variavel de ambiente via Docker API | Nativo Docker | Requer restart do container |
